CONFIRMEDR-01 · Auth trong layout.tsx — plan đúng hướng nhưng thiếu một cảnh báo quan trọng
Plan đề xuất move auth vào src/app/layout.tsx. Đây đúng hướng theo Next.js 16. Tuy nhiên, official Next.js docs cảnh báo: Layouts không re-render khi navigate trong cùng subtree. Nghĩa là auth check trong root layout chỉ chạy lần đầu — các route thay đổi sau đó không trigger lại. Cần thêm auth check trực tiếp ở từng page.tsx hoặc dùng Data Access Layer (DAL) pattern.
Cần bổ sung vào plannextjs.org/docs/app/guides/authentication
Next.js docs: "Pages can be rendered without layouts — adding auth check only to layout is a security vulnerability." Auth phải đặt tại page.tsx hoặc DAL functions, không chỉ layout.
CRITICAL — Plan saiR-02 · vault.secrets KHÔNG cho phép ALTER TABLE trực tiếp
Plan thêm ALTER TABLE vault.secrets ADD CONSTRAINT uq_secret_name UNIQUE (name); vào migration. Đây là sai hoàn toàn. Theo Supabase Vault docs và GitHub issues: vault là protected schema — bạn không có quyền ALTER TABLE vault.secrets. Lệnh này sẽ fail với ERROR: permission denied for table secrets.
Sửa ngaygithub.com/supabase/vault/issues/33
Thực tế: vault.secrets đã có index secrets_name_idx là partial unique index trên name IS NOT NULL. Không thể thêm constraint mới. ON CONFLICT phải dùng: ON CONFLICT ON CONSTRAINT secrets_name_idx — nhưng đây cũng là bug đã report (index partial không dùng được với ON CONFLICT theo cách thông thường). Giải pháp đúng: dùng vault.update_secret() nếu secret đã tồn tại, hoặc upsert qua RPC wrapper riêng.

Critical · Phase 4T-01 · Bypass auth vẫn còn nếu auth chỉ check trong root layout
Scenario: Attacker gửi request đến /dashboard/settings với RSC header RSC: 1. Next.js partial rendering sẽ trả về chỉ nội dung của page segment, bỏ qua layout. Auth check ở app/layout.tsx không được chạy. Attacker thấy nội dung protected mà không cần auth.
Expected: 401. Actual: content leaked. Plan hiện tại chỉ nói "move auth vào layout" — chưa đủ. Cần auth check tại từng page.tsx hoặc DAL.
Critical · Phase 5T-02 · vault upsert pattern — ON CONFLICT sẽ fail ở runtime
Scenario: Migration chạy ALTER TABLE vault.secrets ADD CONSTRAINT... → fail với permission error, migration bị rollback. Sau đó save_user_secret RPC dùng ON CONFLICT (name) DO UPDATE → fail vì không có UNIQUE constraint trên non-partial index. Race condition vẫn tồn tại.
Toàn bộ fix F-009 collapse ở production. Cần approach khác hoàn toàn — xem Solution Optimizer.
High · Phase 6T-03 · Verification test CVE: thiếu test case quan trọng nhất
Plan verification: "Cố tình gửi request kèm header x-middleware-subrequest để test bypass". Đây là test đúng hướng nhưng chưa đủ. Cần test thêm RSC partial render bypass (header RSC: 1), và test server action endpoint trực tiếp không qua UI, vì proxy.ts không intercept server actions nếu config sai.
Bổ sung 2 test cases: (1) curl với -H "RSC: 1" vào protected routes, (2) POST trực tiếp đến /api/server-actions không kèm session cookie.
High · Phase 2T-04 · Handle @username: chặn mà không resolve sẽ break UX cho 30% users
Plan tạm thời chặn handle và hiện thông báo "nhập UC ID". Nhưng YouTube đang push users dùng handle (@username) thay vì UC ID ngày càng nhiều. Users copy URL từ YouTube → nhận được @handle → bị reject. Frustration cao, drop-off rate cao.
Nên ưu tiên auto-resolve handle → UC ID qua YouTube Search API (forHandle param) ngay từ đầu, không phải "nếu cần thiết". Blocked users = churned users.
