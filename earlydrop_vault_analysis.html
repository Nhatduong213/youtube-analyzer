F-001 · CriticalRoot cause confirmed
Silent crash do dot-notation vault query
Supabase JS client không support supabase.from('vault.decrypted_secrets'). Thay vì throw error, nó return object rỗng hoặc gây undefined access ở bước tiếp theo. Không có try/catch bao quanh → function crash im lặng, không có error log nào sau "Direct invoke".
// Bug: Supabase JS không hiểu dot notation const { data, error } = await supabase .from('vault.decrypted_secrets') // → undefined behavior .select('decrypted_secret') .eq('name', keyRef) .single(); // error = null nhưng data = null → downstream crash
F-002 · CriticalSecurity
RPC function thiếu GRANT — ai có thể gọi get_secret()?
SQL được đề xuất tạo function với SECURITY DEFINER nhưng không có REVOKE ALL ON FUNCTION + GRANT EXECUTE TO authenticated. Theo mặc định Postgres cấp EXECUTE cho PUBLIC. Anon user có thể gọi get_secret('stripe_key') qua REST API nếu RLS không chặn.
-- Thiếu trong SQL đề xuất: REVOKE ALL ON FUNCTION public.get_secret(text) FROM PUBLIC; GRANT EXECUTE ON FUNCTION public.get_secret(text) TO service_role;
F-003 · CriticalObservability
Không có error boundary — mọi lỗi sau "Direct invoke" đều im lặng
Phương án đề xuất fix đúng chỗ query vault nhưng không giải quyết vấn đề gốc: function thiếu top-level try/catch. Nếu RPC thành công nhưng API key sai, hoặc downstream service timeout, crash tiếp tục im lặng với 53ms exit.

F-004 · High
get_secret() trả về NULL khi secret không tồn tại — không phân biệt "không có" vs lỗi
SQL function dùng SELECT INTO, nếu không tìm thấy secret thì secret_value = NULL. Caller nhận NULL không có cách phân biệt "secret không tồn tại" với "vault query bị lỗi". Nên raise exception.
F-005 · High
secret_name là user-controlled — SQL injection qua RPC parameter
Nếu keyRef đến từ bất kỳ user input nào (header, body, env), và function SQL không dùng parameterized query đúng cách, có nguy cơ injection. Cần audit nguồn của keyRef.
F-006 · High
53ms CPU budget — RPC thêm round-trip latency
EarlyDrop crash ở 53ms. Thêm RPC call đến Postgres (~5-15ms) có thể đẩy latency lên nhưng quan trọng hơn là nếu Supabase cold-start, RPC có thể timeout hoàn toàn. Cần set timeout riêng.
F-007 · High
keyRef undefined — không validate trước khi query
Nếu keyRef là undefined hoặc empty string, RPC sẽ được gọi với { secret_name: undefined }. Postgres nhận NULL, trả về NULL, không có lỗi. Function tiếp tục với API key null.
Medium
F-008 · Medium
vaultErr không được check đúng cách sau RPC
Code mẫu đề xuất chỉ đổi query method, không thay đổi error handling. Nếu vaultErr !== null sau RPC, cần explicit throw/return để tránh tiếp tục với null secret.
F-009 · Medium
Function name get_secret() quá generic — namespace conflict
Nếu project có nhiều Supabase functions hoặc extension, public.get_secret có thể conflict. Nên đặt tên vault_get_secret hoặc prefix rõ ràng hơn.
F-010 · Medium
Không có cache cho vault secret — query mỗi invocation
Trong hourly-tracker, secret thường không đổi. Query vault mỗi lần chạy thêm latency và load không cần thiết. Nên cache trong memory hoặc dùng environment variable khi deploy.

Root cause analysis
Symptom: Function exit ở 53ms sau log "Direct invoke", không error log.

Root cause: Supabase JS client silently ignores dot-notation schema, trả về query builder không hợp lệ. Bước .single() hoặc property access tiếp theo throw, nhưng không có catch nên Edge Runtime terminate function im lặng.

Depth: Design bug — thiếu defensive programming pattern ở tầng integration. Đây là lỗi đã gặp trước đó trong conversation, nghĩa là pattern vault access chưa được chuẩn hóa trong codebase.
Phòng ngừa
1. Lint rule: cấm .from() với string chứa dấu chấm trong Supabase client.
2. Integration test: mock vault query và assert function không crash khi vault unavailable.
3. Standard: mọi Edge Function phải có top-level try/catch — enforce qua code review template.
