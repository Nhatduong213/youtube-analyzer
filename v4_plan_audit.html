Critical Mới CVE-2025-55182 React2Shell (CVSS 10.0) — version target sai, exploit nhắm trực tiếp vào Server Actions ⌃

Live Researcher Công bố ngày 3/12/2025, khai thác thực tế từ 5/12/2025 bởi nhiều nhóm APT (theo Microsoft, Wiz, Datadog). Lỗ hổng nằm trong React Server Components "Flight" protocol — attacker gửi một HTTP request duy nhất với payload được craft sẵn để thực thi code tùy ý trên server, không cần xác thực.

Imaginative Tester V4 plan có hai vấn đề chồng chéo: (1) Verification plan hướng dẫn test bằng cách gửi POST với header Next-Action không cookie — đây chính xác là vecto khai thác của React2Shell. Một developer chạy test này trên server chưa patch = vô tình thực hiện exploit. (2) Kế hoạch mở rộng Server Actions (YouTube handle, channel upsert) mà không đề cập React2Shell — tăng bề mặt tấn công đúng vào lúc rủi ro cao nhất.

Imaginative Tester Kịch bản thực tế: nếu team deploy với next@15.2.3 (đúng theo kế hoạch V4), attacker có thể RCE trước khi bất kỳ auth check nào trong verifySession() chạy. DAL hoàn toàn vô nghĩa khi server đã bị chiếm.

Solution Optimizer — Immediate
Chạy npx fix-react2shell-next để tự động detect và bump version đúng. Hoặc pin thủ công: npm install next@14.2.35 (cho 14.x) hoặc next@15.2.8 (cho 15.2.x). Rotate toàn bộ secrets sau khi patch vì server có thể đã bị truy cập. Thêm vào Verification Plan: chạy npx fix-react2shell-next --check để verify trước khi deploy.

Nguồn: react.dev/blog (3/12/2025), nextjs.org/blog/CVE-2025-66478, nextjs.org/blog/security-update-2025-12-11 — fetched 23/5/2026

High Mới Upgrade từ 14→15 mang theo breaking change đồng bộ async: cookies(), headers(), params() phải được await ⌃

Live Researcher Next.js 15 breaking change: cookies(), headers(), draftMode(), params, searchParams từ synchronous chuyển sang async. Code cũ const cookieStore = cookies() → phải đổi thành const cookieStore = await cookies().

Imaginative Tester Hàm verifySession() trong DAL mới (Priority 2 kế hoạch) rất có thể đọc session từ cookie. Nếu dùng cú pháp sync, Next.js 15 sẽ show deprecation warning trong dev, nhưng Next.js 16 thì crash hoàn toàn. Upgrade path: npm install next@latest sẽ nhảy thẳng lên 16.x — nếu không fix async trước sẽ phá vỡ toàn bộ auth flow.

Imaginative Tester Kế hoạch ghi "npm install next@latest đảm bảo lên bản vá 14.2.25 hoặc 15.2.3+" — nhưng @latest hiện tại trỏ vào Next.js 16.x, không phải 14 hay 15. Nếu project đang dùng 14.x mà chạy lệnh này, sẽ nhảy 2 major version.

Solution Optimizer
Pin rõ version thay vì dùng @latest. Nếu muốn giữ 14.x: npm install next@14.2.35. Nếu migrate lên 15: chạy codemod async API trước — npx @next/codemod@latest next-async-request-api . — rồi kiểm tra lại DAL. Không dùng next@latest mà không hiểu rõ current latest là version bao nhiêu.

Nguồn: nextjs.org/docs/app/guides/upgrading/version-15, nextjs.org/docs/app/guides/upgrading/version-16 — fetched 23/5/2026

High Vault EXCEPTION block: vault.create_secret có thể không fire unique_violation nếu name là NULL ⌃

Live Researcher Constraint trên vault.secrets.name là partial unique index secrets_name_idx với điều kiện WHERE name IS NOT NULL (confirmed từ GitHub supabase/vault issue #33). Điều này có nghĩa: nếu p_name IS NULL, không có unique constraint nào được kích hoạt — vault.create_secret(p_secret, NULL) không bao giờ throw unique_violation.

Imaginative Tester Kịch bản: RPC save_user_secret được gọi với p_name = NULL nhiều lần. EXCEPTION handler không bao giờ fire → tạo vô số secrets không tên, không thể update. Logic upsert hoàn toàn bypassed. Kế hoạch V4 không có validation cho p_name IS NOT NULL.

Solution Optimizer
Thêm guard ngay đầu RPC: IF p_name IS NULL THEN RAISE EXCEPTION 'p_name cannot be null'; END IF;. Hoặc thêm NOT NULL constraint vào parameter signature. Test case bắt buộc: gọi RPC với p_name = NULL và xác nhận trả lỗi rõ ràng.

Nguồn: github.com/supabase/vault/issues/33 (July 2024)

Medium — technical debt
Medium EXCEPTION block dùng sai scope: BEGIN...EXCEPTION phải là sub-block lồng trong outer block của hàm ⌃

Imaginative Tester Code trong kế hoạch V4 viết:

BEGIN
PERFORM vault.create_secret(...);
EXCEPTION WHEN unique_violation THEN
SELECT id INTO v_id ...
END;

Nếu đây là toàn bộ body của một PL/pgSQL function, cú pháp là hợp lệ. Nhưng nếu block này được lồng bên trong outer function (ví dụ function đã có BEGIN...END ở ngoài), developer cần đảm bảo EXCEPTION handler là inner block riêng, không phải outer block — vì outer block với EXCEPTION làm toàn bộ transaction rollback, không chỉ savepoint.

Imaginative Tester Thêm: trong EXCEPTION handler, câu SELECT id INTO v_id FROM vault.secrets WHERE name = p_name LIMIT 1 có thể trả NULL nếu secret bị xóa giữa hai bước (TOCTOU). vault.update_secret(NULL, ...) sẽ fail với unhandled exception.

Solution Optimizer
Thêm guard: IF v_id IS NULL THEN RAISE EXCEPTION 'Secret disappeared during upsert'; END IF; sau SELECT. Đảm bảo EXCEPTION block là inner sub-block, không phải top-level block của function.
Medium Rate limit 1 req/5s thiếu cơ chế lưu trạng thái: không có persistence layer được chỉ định ⌃

Imaginative Tester Kế hoạch ghi "rate-limit server action 1 request/5s/user" nhưng không chỉ định persistence layer. Server Actions của Next.js là stateless — mỗi request là một invocation mới. Rate limit phải được lưu ở Redis, Upstash, hoặc Supabase table. Nếu deploy multi-instance (scaling), in-memory rate limiting sẽ bị bypass hoàn toàn vì mỗi instance có state riêng.

Imaginative Tester Kịch bản bypass: attacker gửi 10 request/5s bằng cách distribute qua nhiều instances (round-robin load balancer). Mỗi instance chỉ thấy 1 req và không chặn.

Solution Optimizer
Dùng Upstash Redis + @upstash/ratelimit (sliding window, 1 req/5s per userId). Đây là pattern phổ biến với Next.js Server Actions. Thay thế: tạo bảng rate_limit_log trong Supabase với index trên (user_id, created_at) và query đếm request trong 5 giây gần nhất.
